# Doria Brand Positioning

> Documentation role: supporting design note.
> Source-of-truth hierarchy: `docs/doria-end-to-end-plan.md` owns future sequencing; accepted `docs/decisions/*.md` files own topic-level decisions. This note is subordinate to both.

## Core rule

Doria should stand on its own.

## Public description

Doria is a compiled programming language for building native applications, command-line tools, services, games, and systems software with expressive syntax, strong static typing, safe defaults, and modern concurrency.

## Do say

- Doria is expressive.
- Doria is compiled.
- Doria targets native executables.
- Doria is suitable for tools, services, desktop apps, games, future FFI/native library work, and future raylib bindings.
- Doria has safe defaults and strong static checking.
- Doria starts with a Rust bootstrap compiler and should grow toward self-hosting.

## Avoid saying in brand-facing copy

- Doria is PHP but compiled.
- Doria is PHP++.
- Doria is just a PHP output tool.
- Doria is primarily PHP-like.
- Doria should be understandable by a PHP interpreter.

## Allowed technical context

- Doria's syntax is familiar to developers coming from PHP-like and C-like languages.
- Doria may provide PHP migration tooling.
- Doria may have a PHP compatibility backend.
- PHP influenced some design choices, but it does not define Doria's identity.

## Brand analogy

Perl influenced PHP, but PHP became its own language. PHP influenced Doria, but Doria should become its own language.
