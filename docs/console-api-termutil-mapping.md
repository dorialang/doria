# TermUtil → Doria `Console`: API mapping reference (input to the future `std::term` decision)

**Source:** `atatusoft-ltd/termutil@main`, read directly from source. This note maps TermUtil's actual inventory onto the planned `std::term` `Console` surface (plan §9, Stage 46), with a disposition per item: **keep** (same concept, Doria spelling), **redesign** (concept survives, mechanism must change), or **defer/relocate** (not `Console` v1 scope).

## Structural validation

TermUtil's own layering already matches the plan's: `IO/Console` (terminal control) is separate from `UI/Windows` (widget layer: `Window`, `BorderPack`, alignment enums, padding). The plan draws the stdlib line between exactly those two layers (§14: widget frameworks are userland/engine territory), so the split carries over as-is: `IO/Console` informs `Console`; `UI/Windows` informs the *ported engines*, not std.

## `Cursor` — dispositions

`hide`/`show`, `moveTo`, `moveUp/Down/Left/Right($amount)`, `clearLine`, `clearLineFromCursor` → all **keep** as `Console` members (or `Console`-owned cursor namespace per above). `enableBlinking`/`disableBlinking` → **keep, capability-gated** (not universally supported; the capability model absorbs this instead of the user).

## The one mandatory redesign: `Color`

TermUtil's `Color` is a PHP backed enum **whose backing values are the ANSI escape strings themselves** (`case BLUE = "\033[0;34m"`). This is precisely why TermUtil cannot be portable: the Unix implementation lives inside the public type. In Doria, `Color` (and `Style`) are pure enums that mean *colors* — `Color::Blue` — and each platform backend renders them (VT sequences where enabled, console attributes on legacy Windows). **Future-decision constraint: no `std::term` public type may carry escape sequences, VT fragments, or platform encodings in its values or API.** This single change is what turns the same API portable.

## Also present in TermUtil, already resolved elsewhere

- `Interfaces/ComparableInterface`, `EquatableInterface` → Doria core already plans `Comparable<T>`, `Equatable<T>` (plan §9) — TermUtil independently wanting them is nice confirmation.
- Observer traits / static observables → superseded by `Console::pollKey()`/event polling; no observer machinery in std::term v1.
- `Functions.php`, `Constants.php` grab-bags → dissolved into the charter-named surface.
